## Task
Improve `crates/uu/src/runner.rs` based on vibecheck findings.

## Findings (top 20)
- [medium] edge-cases — `crates/uu/src/runner.rs:98-123`
  - ANSI escape codes are unconditionally emitted in `style()` and `run_steps()`. When output is piped to a file or another program (non-TTY), the escape codes pollute the output. No `--color` flag or `isatty` check exists.
  - Suggested fix: Add an `isatty(stderr)` check (e.g. via `std::io::stderr().is_terminal()` on Rust 1.70+) to conditionally emit ANSI codes. Optionally expose a `--color=auto|always|never` global flag.
- [medium] input-validation — `crates/uu/src/runner.rs:112-124`
  - `run_steps` executes subprocess programs directly from `Step.program` using `Command::new`. While the program names are currently hardcoded (safe), the `./gradlew` case executes a script from the current working directory. If a user runs `uu build` in a cloned repository containing a malicious `gradlew` script, arbitrary code executes with the user's full permissions. This is inherent to Gradle's design, but `uu` offers no warning about it, unlike `gradle` itself which documents this trust boundary.
  - Suggested fix: When executing relative-path programs like `./gradlew`, emit a warning (e.g., "running project-local script ./gradlew — ensure you trust this repository") on first use. Consider checking the file's ownership and permissions, or at minimum document this trust boundary in the README.
- [low] edge-cases — `crates/uu/src/runner.rs:32-38`
  - `Step::Display` only quotes arguments containing spaces (`' '`). Arguments containing shell metacharacters like `$`, `"`, `'`, `\`, `|`, `>`, `<`, `;`, `&`, `(`, `)`, newlines, etc. are displayed unquoted, which would be misleading if copy-pasted into a terminal.
  - Suggested fix: Expand the quoting logic to handle all shell-special characters, or use a proper shell-quoting function (e.g. `shell_escape` crate, or just quote any arg that isn't purely alphanumeric/dashes/dots/slashes).

## Notes
- Keep behavior the same unless the finding explicitly calls for a behavior change.
- Prefer small, focused changes.
