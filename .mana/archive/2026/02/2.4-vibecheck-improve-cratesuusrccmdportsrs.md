## Task
Improve `crates/uu/src/cmd/ports.rs` based on vibecheck findings.

## Findings (top 20)
- [medium] graceful-degradation — `crates/uu/src/cmd/ports.rs:106-117`
  - `list_listeners()` depends on `lsof` which is macOS/Linux-specific. On Windows (or systems without `lsof`) this silently fails or gives a confusing error. The `command_on_path` utility exists in the detect crate but isn't used here to check for `lsof` and suggest alternatives.
  - Suggested fix: Check for `lsof` availability first and give a clear error on unsupported platforms. Consider `ss` as a Linux fallback. At minimum, document the platform requirement.
- [medium] input-validation — `crates/uu/src/cmd/ports.rs:62-82`
  - `uu ports -k <port>` sends SIGTERM to processes without any confirmation prompt. Killing a process is destructive and irreversible — a typo (`uu ports 5432 -k` instead of `uu ports 5432`) could kill a production database. The `-k` flag is a single character away from being accidentally invoked, and there's no "are you sure?" gate, no `--force` requirement, and the short flag `-k` makes it easy to combine accidentally with other flags.
  - Suggested fix: Add a confirmation prompt before killing (e.g., "Kill node (pid 12345) on :3000? [y/N]") that can be bypassed with `--yes` or `-y` for scripting. Alternatively, require `--kill` (long form only) to reduce accidental invocations, and consider showing the process info before asking for confirmation.
- [low] graceful-degradation — `crates/uu/src/cmd/ports.rs:91-101`
  - `list_listeners()` suppresses stderr from `lsof` (via `Stdio::null()`) and silently produces an empty list if `lsof` fails (non-zero exit or not found). On Linux, `lsof` requires root for some process info, and on systems without `lsof` (e.g., containers), the command silently returns no results. The user sees "no listening TCP ports found" which is misleading — it should distinguish between "nothing listening" and "couldn't check."
  - Suggested fix: Check the exit status of `lsof`. If it fails or isn't found, print a warning explaining that `lsof` is required and may need elevated permissions. On Linux, consider falling back to parsing `/proc/net/tcp` which doesn't require `lsof`.
