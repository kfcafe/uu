## Task
Improve `crates/uu/src/runner.rs` based on vibecheck findings.

## Findings (top 20)
- [high] error-handling — `crates/uu/src/runner.rs:80-82`
  - `run_steps` calls `process::exit()` on command failure, which immediately terminates without unwinding the stack. This bypasses all Rust destructors (Drop impls), meaning temp files, locks, or any RAII cleanup won't run. It also makes the function untestable for failure paths — you can't write a unit test for a function that kills the process.
  - Suggested fix: Return an error with the exit code instead: `return Err(anyhow::anyhow!("command failed with exit code {}", status.code().unwrap_or(1)))`. Handle the exit code in `main()` where `process::exit()` is appropriate.
- [medium] resource-cleanup — `crates/uu/src/runner.rs:78-80`
  - `process::exit()` is called directly when a subprocess fails, bypassing all Rust destructors (Drop implementations). This skips cleanup for any RAII guards, temporary files, buffered I/O flushes, or lock files that may be held. Currently the tool doesn't hold resources at exit time, but this creates a latent risk — any future code adding cleanup logic (temp dirs, lock files, progress state) will silently fail to run.
  - Suggested fix: Propagate the exit code upward via `Result` or a custom error type (e.g., `anyhow::bail!("command failed with exit code {}", code)`) and let `main()` call `process::exit()` as the single exit point. Alternatively, use `std::process::ExitCode` as the return type of `main()`.
- [low] comprehensibility — `crates/uu/src/runner.rs:69-73`
  - ANSI color codes are passed as raw strings (`"31"`, `"32"`, `"33"`, `"36"`) throughout the codebase. Readers must know the ANSI spec to understand that `"31"` means red, `"32"` green, etc.
  - Suggested fix: Replace with named constants or an enum: `const RED: &str = "31"; const GREEN: &str = "32"; const YELLOW: &str = "33"; const CYAN: &str = "36";`
- [low] single-responsibility — `crates/uu/src/runner.rs:1-135`
  - `runner.rs` handles three unrelated concerns: project detection (`detect_project`), command execution (`run_steps`, `step`, `append_args`, `Step`), and terminal formatting (`style`). As the tool grows, this module will become a grab-bag.
  - Suggested fix: This is fine at the current size (~135 lines), but worth noting for future growth. If more formatting or detection logic is added, split into `detect.rs` (project detection), `exec.rs` (step running), and `ui.rs` (terminal output).
- [low] input-validation — `crates/uu/src/runner.rs:59-72`
  - User-provided extra arguments (passed via `--`) are forwarded directly to subprocess commands without any validation. While this is by design and the arguments are passed as separate argv entries (not through a shell), there's no sanitization to prevent passing dangerous flags to underlying tools. For example, `uu install -- --root /` for pip could install packages system-wide, or `uu clean` (if it accepted extra args) could be misused. Since the user is already trusted (they're running the CLI), this is low severity, but there's no mechanism to warn about potentially destructive passthrough flags.
  - Suggested fix: Document the passthrough behavior explicitly in `--help` output for each command. Consider adding a warning in `--dry-run` mode when extra args are passed that could be destructive (e.g., args containing paths or `--root`/`--prefix`).
