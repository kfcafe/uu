## Task
Improve `crates/uu/src/cmd/clean.rs` based on vibecheck findings.

## Findings (top 20)
- [medium] consistency — `crates/uu/src/cmd/clean.rs:15-101`
  - `clean::execute` manually reimplements command execution (spawning processes, checking status, printing styled output) instead of using the shared `run_steps()` infrastructure. This duplicates error handling and output formatting patterns from `runner.rs`, and means any improvements to `run_steps` (e.g., signal handling, logging) won't apply to `clean`.
  - Suggested fix: Refactor `clean` to use `run_steps()` for the native clean command, then handle artifact directory removal separately. The native clean part can reuse `Step` + `run_steps()` directly.
- [medium] edge-cases — `crates/uu/src/cmd/clean.rs:63-75`
  - Before calling `fs::remove_dir_all()`, there is no check whether the artifact directory path is a symlink. If an attacker (or a compromised dependency) places a symlink at a known artifact directory name (e.g., `target/`, `node_modules/`, `build/`) pointing outside the project, `remove_dir_all` behavior with symlink targets varies by platform and Rust version. While modern Rust (1.63+) removes the symlink itself rather than following it, the `dir_size()` call on line 67 DOES traverse into symlink targets (via `path.is_dir()` which follows symlinks), reporting misleading sizes and potentially hanging on circular symlinks.
  - Suggested fix: Before processing each artifact directory, check `std::fs::symlink_metadata()` and skip (with a warning) if the path is a symlink. This protects both the size calculation and the removal, and makes the behavior explicit rather than relying on platform-specific `remove_dir_all` semantics.
- [low] edge-cases — `crates/uu/src/cmd/clean.rs:38-56`
  - For Cargo projects, `native_clean_cmd` returns `cargo clean` (which removes `target/`), and then `artifact_dirs()` returns `["target"]` — so the code tries to also `rm -rf target/` after `cargo clean` already deleted it. This is harmless (the `exists()` check skips it) but means `total_freed` is 0 and the output says "nothing to clean" instead of showing what `cargo clean` freed.
  - Suggested fix: Either skip `artifact_dirs` removal for ecosystems that have a native clean command, or calculate and display the size before running the native command.
- [low] edge-cases — `crates/uu/src/cmd/clean.rs:90-103`
  - `dir_size_inner()` recurses into directories without any depth limit or cycle detection. If the filesystem contains circular symlinks (e.g., `node_modules/a -> node_modules/b`, `node_modules/b -> node_modules/a`), this function will recurse until stack overflow. The `entry.file_type()?.is_dir()` check does not follow symlinks on most platforms, but `path.is_dir()` on line 92 (the initial call) does follow them.
  - Suggested fix: Add a maximum recursion depth or track visited inodes via `metadata().ino()`. Alternatively, use `symlink_metadata()` / `file_type().is_symlink()` to skip symlinks entirely during traversal, since artifact size reporting doesn't need to follow links.
- [low] error-handling — `crates/uu/src/cmd/clean.rs:42-51`
  - When the native clean command (e.g., `cargo clean`, `go clean`) fails, only a warning is printed and execution continues to delete artifact directories. This means if `cargo clean` fails because the project is in a broken state, the tool still tries `fs::remove_dir_all("target/")` which could fail differently or succeed partially. The warning message uses `status.code().unwrap_or(-1)` but `-1` is not a valid exit code and is misleading — it means the process was killed by a signal.
  - Suggested fix: Use `status.code().map_or("signal".to_string(), |c| c.to_string())` for accurate exit reporting. Consider making the native clean failure non-fatal only if artifact directory removal succeeds, or providing a `--force` flag to skip the native command entirely.
