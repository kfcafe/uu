## Task
Improve `crates/uu/src/cmd/ci.rs` based on vibecheck findings.

## Findings (top 20)
- [medium] consistency — `crates/uu/src/cmd/ci.rs:25-60`
  - The Python uv/non-uv tool-detection pattern (check for ruff → check for black/flake8 → bail with install instructions) is duplicated across `ci.rs`, `fmt.rs`, and `lint.rs`, each with slightly different tools and messages. The two `uv: true` and `uv: false` branches within each file also repeat 80% of their logic, differing only in the command prefix.
  - Suggested fix: Extract shared helpers in `runner.rs` like `python_fmt_steps(uv: bool)`, `python_lint_steps(uv: bool)`, and `python_ci_steps(uv: bool)` that encapsulate the ruff/black/flake8 detection and error messaging.
- [medium] test-coverage — `crates/uu/src/cmd/ci.rs:25-60`
  - Python CI steps are untested because they depend on `command_on_path("ruff")` / `command_on_path("black")` which query the real system PATH. The test suite has no way to exercise Python CI/fmt/lint paths in a controlled environment. This means the ruff/black/flake8 fallback logic has zero automated coverage.
  - Suggested fix: Extract the tool-detection into a trait or accept a tool-checker function parameter so tests can inject fake availability. Alternatively, add unit tests that directly call the internal `steps()` function with known PATH configurations (using `std::env::set_var` in a serial test).
- [low] edge-cases — `crates/uu/src/cmd/ci.rs:23-25`
  - The Go format check uses `step("sh", &["-c", "test -z \"$(gofmt -l .)\""])` which shells out via `sh -c`. This is the only command in the entire codebase that uses shell interpretation rather than direct execution. While the command is hardcoded (no user input reaches the shell string), this is an inconsistency that creates a different failure mode — if a filename contains special shell characters, it could cause unexpected behavior in the `gofmt -l .` output that `test -z` evaluates.
  - Suggested fix: Consider running `gofmt -l .` directly and checking if stdout is empty in Rust code, similar to how `lsof` output is parsed in `ports.rs`. This avoids the shell entirely and gives better error messages (can list which files are unformatted).
