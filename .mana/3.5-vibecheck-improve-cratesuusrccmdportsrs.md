## Task
Improve `crates/uu/src/cmd/ports.rs` based on vibecheck findings.

## Findings (top 20)
- [medium] input-validation — `crates/uu/src/cmd/ports.rs:59-88`
  - `kill_port` sends SIGTERM to processes without any user confirmation. A user running `uu ports --kill 3000` could accidentally kill a critical process if the port was reassigned or if `lsof` output is stale. There is no interactive confirmation prompt, no check that the process belongs to the current user, and no dry-run equivalent for the kill operation (unlike other destructive commands in the tool which support `--dry-run`).
  - Suggested fix: Add an interactive confirmation prompt showing the process name, PID, and user before sending the kill signal (e.g., "Kill node (pid 12345, user asher) on :3000? [y/N]"). Also consider adding `--dry-run` support to the `ports` subcommand for consistency with other destructive operations like `clean`.
- [low] edge-cases — `crates/uu/src/cmd/ports.rs:155-159`
  - `parse_lsof_line` uses `fields.len().wrapping_sub(2)` to index from the end when `(LISTEN)` is the last field. If `fields.len()` is 0 or 1, `wrapping_sub(2)` wraps to `usize::MAX`, and `fields.get()` returns `None` which is safely handled. However, using `wrapping_sub` for defensive arithmetic in a parser is a code smell — it obscures the intent and could mask logic errors in future changes.
  - Suggested fix: Use `fields.len().checked_sub(2)` and explicitly handle the `None` case, or restructure the parsing to use `fields.get(fields.len() - 2)` only when `fields.len() >= 2` is already guaranteed by the earlier length check.
- [low] error-handling — `crates/uu/src/cmd/ports.rs:98-109`
  - `list_listeners` runs `lsof -iTCP -sTCP:LISTEN -nP` but does not handle the case where lsof requires root/sudo to see all listening sockets. On macOS and some Linux configurations, non-root users may not see all listeners. The code warns on non-zero/non-1 exit codes but doesn't inform the user that results may be incomplete without elevated privileges, which could lead to `uu ports --kill 3000` reporting "nothing listening" when a root-owned process is actually using the port.
  - Suggested fix: After a successful lsof invocation, check if the user is non-root and add a hint like "note: some listeners may be hidden — run with sudo for complete results" when the result set is empty or when the user is querying a specific port that shows no results.

## Notes
- Keep behavior the same unless the finding explicitly calls for a behavior change.
- Prefer small, focused changes.
