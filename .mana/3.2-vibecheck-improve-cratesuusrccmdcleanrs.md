## Task
Improve `crates/uu/src/cmd/clean.rs` based on vibecheck findings.

## Findings (top 20)
- [medium] input-validation — `crates/uu/src/cmd/clean.rs:37-96`
  - TOCTOU (time-of-check-time-of-use) race condition between the symlink check on line 37 (`path.symlink_metadata()`) and the actual deletion on line 96 (`fs::remove_dir_all`). A local attacker with write access to the project directory could replace a legitimate artifact directory with a symlink between the check and the removal. While `remove_dir_all` itself doesn't follow symlinks on modern Linux (it uses `openat`), on some platforms or older Rust versions the behavior may differ. The safe approach is to use `openat`-based removal or re-check immediately before deletion.
  - Suggested fix: Use `std::fs::remove_dir_all` with an `openat`-based approach, or at minimum re-verify the symlink status immediately before deletion. Consider using the `remove_dir_all` crate which provides CVE-resistant recursive removal that is safe against symlink races on all platforms.
- [low] performance — `crates/uu/src/cmd/clean.rs:15-17`
  - `runner::detect_project()` calls `env::current_dir()` internally, then `execute()` calls `env::current_dir()` a second time on line 16. Two syscalls for the same value.
  - Suggested fix: Refactor `detect_project()` to return `(ProjectKind, PathBuf)` or accept an optional directory, so callers that need the path don't syscall twice. Same issue exists in `run.rs` line 79-80.
- [low] single-responsibility — `crates/uu/src/cmd/clean.rs:14-103`
  - The `execute()` function is 90 lines and handles detection, size calculation, native clean command dispatch, artifact directory removal, and summary output — five distinct responsibilities in one function. Every other command module delegates to `runner::run_steps()` but `clean` reimplements its own execution flow.
  - Suggested fix: Extract helpers: `calculate_artifact_sizes()`, `run_native_clean()`, `remove_artifact_dirs()`. This would make the logic easier to test independently and keep `execute()` as a coordinator.
- [low] resource-cleanup — `crates/uu/src/cmd/clean.rs:135-136`
  - `dir_size` silently swallows all errors by calling `dir_size_inner(path).unwrap_or(0)`. If `read_dir` fails due to permission errors or I/O issues, the function reports 0 bytes instead of warning the user. This means `uu clean` could report "nothing to clean" or undercount freed space when the user lacks read permissions on subdirectories, masking real problems.
  - Suggested fix: Propagate or log errors from `dir_size_inner`. At minimum, emit a warning to stderr when a directory can't be read (e.g., "warning: could not read {path}: permission denied") so the user knows the size estimate is incomplete.
- [low] edge-cases — `crates/uu/src/cmd/clean.rs:139-153`
  - `dir_size_inner` uses unbounded recursion to traverse directory trees. Deeply nested directory structures (e.g., a pathological `node_modules` tree with thousands of nesting levels) could cause a stack overflow and crash the process. This is a denial-of-service vector if `uu clean` is run in a directory with adversarial nesting.
  - Suggested fix: Convert to an iterative approach using an explicit stack (e.g., `Vec<PathBuf>`) or cap recursion depth with a reasonable limit (e.g., 256 levels). Alternatively, use the `walkdir` crate which handles this safely.
- [low] graceful-degradation — `crates/uu/src/cmd/clean.rs:51-79`
  - When the native clean command (e.g., `cargo clean`) fails, the code logs a warning but continues to remove artifact directories. If `cargo clean` failed because of a locked build directory, the subsequent `remove_dir_all` might also fail with a less clear error. More importantly, if the native clean partially succeeds, the pre-calculated `artifact_sizes` become inaccurate — the "freed" size reported to the user could be double-counted (native clean already freed some, then `remove_dir_all` frees the rest or the dir is already gone).
  - Suggested fix: Recalculate sizes after native clean runs, or skip manual removal for directories that the native clean is expected to handle (e.g., skip `target/` removal when `cargo clean` succeeds). At minimum, note in the output that the freed size is an estimate.

## Notes
